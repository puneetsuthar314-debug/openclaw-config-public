# 项目隔离技能 (Project Isolation)

## 概述

本技能定义了项目隔离的完整规则。当用户要求开发新功能或新项目时，**必须**在 `projects/` 目录下创建独立的项目文件夹，所有项目相关文件（代码、文档、资源、输出）都存放在该文件夹内，不与其他项目或全局文件混合。

---

## 核心规则

### 规则 1：项目必须隔离

任何涉及"开发"、"写代码"、"做一个"、"创建项目"的任务，都必须在 `projects/<项目名>/` 下独立运行。禁止在 workspace 根目录或其他项目目录下创建项目文件。

### 规则 2：标准目录结构

每个新项目必须包含以下结构：

```
projects/<项目名>/
├── PROJECT.md           # 项目上下文（必须）
├── .context.md          # 工作上下文（自动维护）
├── src/                 # 源代码
├── tests/               # 测试文件
├── docs/                # 项目文档
├── outputs/             # 输出文件（报告、图表等）
├── references/          # 参考资料
└── progress.md          # 进度日志
```

### 规则 3：PROJECT.md 是项目的灵魂

每个项目的 `PROJECT.md` 必须包含以下信息：

```markdown
# <项目名称>

## 目标
<一句话描述项目要做什么>

## 技术栈
<列出使用的语言、框架、数据库等>

## 当前状态
<🟡 进行中 / ✅ 完成 / ⏸️ 暂停 — 简述当前阶段>

## 待办事项
- [ ] <具体可执行的任务>

## 关键决策记录
- <日期>: <决策内容和原因>

## 已知问题
<列出已知但尚未解决的问题>
```

### 规则 4：.context.md 自动维护

每次在项目中工作时，在以下时机更新 `.context.md`：
- 开始工作时（记录"正在做什么"）
- 完成一个阶段时（记录"上次停在哪里"）
- 会话即将结束或压缩时（保存当前状态）

格式：

```markdown
# 当前工作上下文
最后更新: <时间>

## 正在做什么
<当前正在进行的具体工作>

## 上次停在哪里
<上次工作的断点，下次从哪里继续>

## 关键文件
<当前正在编辑或参考的文件列表>
```

### 规则 5：PROJECTS.md 索引必须同步

workspace 根目录的 `PROJECTS.md` 是所有项目的索引。每次创建、完成或暂停项目时，必须同步更新此文件。

---

## 触发条件

当用户的消息符合以下任一模式时，触发项目隔离流程：

1. 明确说"新项目"、"开一个项目"、"创建项目"
2. 要求"开发一个 XXX"、"做一个 XXX"、"写一个 XXX 系统/工具/脚本/应用"
3. 任务预计涉及多个文件和多步骤开发

**不触发**的情况：单文件脚本、一次性查询、修改现有文件、日常对话。

---

## 工作流程

### 流程 A：创建新项目

```
用户: "帮我做一个课程提醒机器人"
```

**Step 1 — 确认项目信息**

向用户确认（如果信息不足）：
- 项目名称（英文短横线命名，如 `course-reminder`）
- 核心目标
- 技术偏好（如有）

如果用户描述已经足够清晰，可以跳过确认直接创建。

**Step 2 — 创建项目脚手架**

```bash
# 创建目录结构
mkdir -p projects/course-reminder/{src,tests,docs,outputs,references}

# 生成 PROJECT.md（根据用户需求填写）
# 生成 .context.md（初始状态）
# 生成 progress.md（空模板）
```

**Step 3 — 更新 PROJECTS.md**

在项目索引中添加新条目。

**Step 4 — 切换到项目目录开始工作**

后续所有代码文件写入 `projects/course-reminder/src/`，文档写入 `docs/`，测试写入 `tests/`。

**Step 5 — 通知用户**

简短告知项目已创建，展示目录结构，然后开始执行。

### 流程 B：切换到已有项目

```
用户: "继续做课程提醒那个项目"
```

**Step 1** — 读取 `PROJECTS.md` 找到项目路径
**Step 2** — 读取 `projects/course-reminder/PROJECT.md` 了解项目全貌
**Step 3** — 读取 `projects/course-reminder/.context.md` 恢复工作状态
**Step 4** — 继续工作

### 流程 C：完成项目

**Step 1** — 更新 `PROJECT.md` 状态为 ✅ 完成
**Step 2** — 更新 `PROJECTS.md` 索引
**Step 3** — 清理 `.context.md`
**Step 4** — 在 `progress.md` 中记录项目总结

## 项目命名规范

项目目录名使用**英文小写 + 短横线**格式：

| 用户描述 | 项目目录名 |
|----------|-----------|
| 课程提醒机器人 | `course-reminder` |
| AI写小说系统 | `novel-forge` |
| 自动备份脚本 | `auto-backup` |
| BTC价格分析 | `btc-analysis` |
| 王者荣耀皮肤爬虫 | `wangzhe-skin-crawler` |

## 与模型路由的配合

创建项目时，根据项目类型自动选择最优模型：

| 项目类型 | 推荐模型 | 触发条件 |
|----------|---------|---------|
| 代码开发 | qwen3-coder-plus | 涉及编写代码、脚本、API |
| 数据分析 | qwen3-max | 涉及数据处理、统计、可视化 |
| 研究报告 | qwen3-max | 涉及调研、分析、报告撰写 |
| 通用项目 | qwen3.5-plus | 其他类型 |

在 `PROJECT.md` 中记录推荐模型，切换到该项目时自动执行 `/model` 命令。

---

## 注意事项

1. **不要在项目目录外创建项目文件。** 所有代码、文档、资源都必须在 `projects/<项目名>/` 内。
2. **不要在不同项目之间共享代码文件。** 如果需要共享，提取为全局脚本放在 `scripts/` 目录。
3. **小任务不需要创建项目。** 单文件脚本、一次性查询等直接执行即可。
4. **项目完成后不要删除。** 保留在 `projects/` 目录中作为归档，更新状态为 ✅ 完成。
