## 2026-03-09: 图片下载陷阱

**问题**：直接 curl/wget 搜索引擎返回的图片 URL，经常下载到 HTML/XML 错误页面（如 `hellfire_wangzhe.jpg` 实际是 XML 文档）。
**原因**：搜索引擎的图片缩略图 URL 有时效性，过期后返回错误页面。
**预防**：
1. 必须使用 `image-search-sender` 技能，禁止手动 curl 图片
2. 发送图片前必须用 `file` 命令验证文件类型
3. 只有 `JPEG image data` 或 `PNG image data` 才是有效图片

## 2026-03-09: delivery-queue 死信循环

**问题**：delivery-queue 中失败的消息无限重试，导致钉钉反复收到"图片加载失败"的占位图。
**原因**：`to` 字段设置为 `"heartbeat"`（不是有效的钉钉 staffId），消息永远无法投递成功。
**预防**：
1. delivery 的 `to` 字段必须是有效的钉钉 staffId（如 `manager20`）
2. 定期检查 delivery-queue/failed 目录，超过 7 天的失败消息自动清理
3. weekly-cleanup 脚本已增加死信队列清理功能

## 2026-03-09: aliyun_code_interpreter 远程沙箱文件不在本地（严重！）
**问题**：使用 `aliyun_code_interpreter` 下载图片到 `/root/.openclaw/workspace/images/caiwenji.jpg`，code_interpreter 返回"成功下载"，但文件实际保存在**阿里云远程沙箱**中，本地服务器上该文件不存在。随后输出 `[DING:IMAGE path="..."]` 标签时，本地找不到文件，导致"图片上传失败"。
**原因**：`aliyun_code_interpreter` 是阿里云百炼提供的远程代码执行环境（MCP），它的文件系统与本地服务器完全隔离。在远程沙箱中执行的文件操作（下载、保存）不会同步到本地。
**预防**：
1. **绝对禁止**使用 `aliyun_code_interpreter` 下载图片或文件到本地路径——它下载的文件只在远程沙箱中
2. 下载图片必须使用本地 `exec` 工具执行 `image_search.py` 脚本（`image-search-sender` 技能）
3. 下载文件必须使用本地 `exec` 工具执行 `curl`/`wget` 命令
4. `aliyun_code_interpreter` 只适合用于数据分析、计算等不需要本地文件落盘的场景

### 教训：发送多张图片时只发了1张（2026-03-09）
- **问题**：用户要求发送5张蔡文姬图片，AI用 image_search.py 下载了5张，但回复中只写了1个 `[DING:IMAGE]` 标签
- **原因**：AI没有意识到需要为每张图片单独写一个 `[DING:IMAGE]` 标签
- **解决**：用户要求几张就必须写几个 `[DING:IMAGE]` 标签，每张图片一个，不能偷懒只发一张

### 教训：用户要求 1 张却发了 5 张（2026-03-09）
- **问题**：用户说"发一张王者荣耀多利亚照片"，AI 却下载并发送了 5 张
- **原因**：AI 机械套用 image-search-sender 技能中"用户要求几张就发几张"的规则，但没有正确理解用户的原始请求中明确说了"一张"
- **解决**：
  1. 必须仔细理解用户原始请求中的数量词（"一张"、"5 张"、"几张"）
  2. 下载图片时 `--limit` 参数要与用户要求的数量一致，不要默认下载 5 张
  3. 用户没说数量时默认 1 张，不要自作主张多发

## 2026-03-10: 模型路由检查未作为前置必做步骤（流程缺陷）
**问题**：在股票分析任务中，虽然 AGENTS.md 有"智能模型路由"章节，但实际执行时跳过了模型路由检查，直接开始搜索和分析。
**根本原因**：
1. "任务分类与处理"和"智能模型路由"是两个独立章节，没有强制关联
2. 任务执行框架（L1/L2/L3）中只提到了复杂度判断，没有明确要求先做模型路由检查
3. 把模型路由当成了"可选优化"，而不是"前置必做步骤"

**流程优化（已更新 AGENTS.md）**：
1. 在"任务分类与处理"章节增加**双重前置检查**要求：
   - 第一维：任务复杂度（L1/L2/L3）→ 决定执行流程
   - 第二维：任务类型（代码/图片/分析/推理）→ 决定模型切换
   - **检查顺序：先判断任务类型切换模型，再判断复杂度执行任务**
2. 在 L1/L2/L3 所有任务流程的第一步都加入"**模型路由检查**"
3. 在"自动切换规则"中增加**执行纪律**说明：模型路由检查是前置必做步骤，不是可选优化
4. 在"切换协议"中增加：任务完成后必须自动切回默认模型
5. 在"质量自检清单"中增加：模型路由是否正确执行

**违规后果**：跳过模型路由检查直接执行任务 = 执行流程缺陷，必须记录到 lessons.md

**核心改进**：模型路由不再是独立章节的"建议"，而是任务执行框架的**强制第一步**

## 2026-03-11: 走势图生成成功但未发送到钉钉群（Subagent 静默失败）

**问题**：用户要求发送翔鹭钨业走势图到钉钉群，图表在本地成功生成，但用户在群里什么都没收到。

**根本原因链**：
1. 主 Agent 将走势图任务委托给了 Subagent
2. Subagent 尝试用浏览器截图但 Chrome CDP websocket 连接失败
3. Subagent 改用 web_fetch + 本地 Python 脚本生成了图表
4. 图表成功保存到 /root/.openclaw/workspace/images/xianglu_002842_chart.png
5. **但 Subagent 遇到 API rate limit，announce completion 失败**
6. 主 Agent 收到 "No reply from agent"，任务静默结束
7. 没有人输出 [DING:IMAGE] 标签，用户什么都没收到

**深层原因**：
1. 主 Agent 对 Subagent 采取了"发射后不管"的策略，没有检查 Subagent 是否成功交付
2. 涉及钉钉图片发送的任务不应委托给 Subagent（Subagent 无法直接发送 [DING:IMAGE] 标签）
3. 缺少"生成文件后必须发送"的强制检查环节

**预防措施（已更新 AGENTS.md）**：
1. 新增"Subagent 失败回退"硬性规则：Subagent 失败后主 Agent 必须自行接管
2. 新增"钉钉交付物最终检查"硬性规则：生成 → 验证 → 发送 → 确认，四步缺一不可
3. 明确规定：涉及图片/文件发送的钉钉任务，不建议委托给 Subagent
4. 新增中间产物检查流程：Subagent 失败后检查 images/tmp/files/ 目录

**正确流程**：
- 收到"发送走势图"→ 主 Agent 直接执行（不委托 Subagent）→ 生成图表 → ls -la 验证文件 → 输出 [DING:IMAGE path="..."] 标签 → 确认发送成功

**执行口诀**：生成 → 验证 → 发送 → 确认。四步缺一不可。
