## 🧠 长篇记忆管理方案（核心技术难点）

### 问题本质：大模型的"金鱼记忆"困境

**限制：**
- 主流 LLM 上下文窗口：2048-32768 token（约几千到几万字）
- 写 10 万字小说，模型必须不断丢弃早期信息
- 导致：人设漂移、设定打架、伏笔失控

**"迷失在中间"效应：**
- 即便上下文足够大（如 Claude 3 的 200K），模型对开头和结尾关注度高，中间部分常被忽略

---

### 解决方案：分层记忆管理

借鉴人类记忆机制 — 不试图记住每一个字，而是提取关键信息存入"长期记忆"

```
┌──────────────────────────────────────────────┐
│          写作过程中的记忆分层                 │
├──────────────────────────────────────────────┤
│  工作记忆 ←→ 长期记忆                        │
│  (当前章节)    (人设/主线/伏笔台账)           │
│       ↑              ↑                       │
│       └──── 召回关键信息 ────┘               │
│       └──── 存储重要进展 ────┘               │
└──────────────────────────────────────────────┘
```

### 三种技术实现路径

#### 方案一：动态记忆管理框架（MemGPT）
**来源：** 加州伯克利大学开源

**核心逻辑：**
```python
class MemGPT:
    def generate_response(self, user_input):
        # 1. 将用户输入加入工作记忆
        self.working_memory.append(user_input)
        
        # 2. 元认知控制判断是否需要记忆操作
        action = self.controller.decide(self.working_memory)
        
        if action == "STORE":
            # 提取关键信息存入长期记忆
            relevant_info = extract_key_info(self.working_memory)
            self.long_term_memory.store(relevant_info)
            self.working_memory = self.working_memory[-512:]  # 保持工作记忆大小
            
        elif action == "RECALL":
            # 从长期记忆检索相关上下文
            context = self.long_term_memory.query(user_input)
            self.working_memory.extend(context)
        
        # 3. 基于完整上下文生成响应
        return self.llm.generate(self.working_memory)
```

**优点：** 自动化记忆管理，无需人工干预
**缺点：** 实现复杂，需要额外框架支持

---

#### 方案二：工程化写作框架（实操流）⭐推荐

**核心思路：** 分步管理 + 台账更新

**实施步骤：**
1. **创建记忆文件体系**
   - `characters.md` — 角色档案（外貌/性格/背景/动机）
   - `world.md` — 世界观设定（地理/历史/规则）
   - `plot.md` — 主线剧情（核心冲突/关键节点）
   - `foreshadowing.md` — 伏笔台账（埋设章节/回收章节/状态）
   - `timeline.md` — 时间线（事件顺序）

2. **每章写作流程**
   - 写作前：从记忆文件加载相关上下文（人设 + 世界观 + 前情提要）
   - 写作中：AI 基于完整上下文创作
   - 写作后：提取新信息更新记忆文件（新角色/新设定/新伏笔）

3. **关键信息提取规则**
   - 新角色登场 → 更新 `characters.md`
   - 新地点/设定 → 更新 `world.md`
   - 伏笔埋设/回收 → 更新 `foreshadowing.md`
   - 重大剧情转折 → 更新 `plot.md`

**优点：** 简单可控，无需复杂框架
**缺点：** 需要人工审核更新内容

---

#### 方案三：多智能体协作框架（StoryWriter）

**核心思路：** 多个 AI 分工协作

- **作者 Agent** — 负责创作正文
- **编辑 Agent** — 负责检查逻辑漏洞
- **记忆管理员 Agent** — 负责更新记忆文件
- **世界观顾问 Agent** — 负责设定一致性检查

**优点：** 质量高，自动化程度高
**缺点：** 成本高（多次 API 调用），实现复杂
