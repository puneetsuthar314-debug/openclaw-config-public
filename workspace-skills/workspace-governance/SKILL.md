# workspace-governance — 工作区自治治理

## 概述
你是自己工作区的管理者。这个技能定义了你如何**用自己的能力**（读文件、写文件、对比、推理）来维护工作区的健康，而不是依赖外部脚本。

## 触发时机
以下情况你应该执行自检：
1. 收到定时任务的"执行 workspace-governance 技能"指令时
2. 用户反馈你"失忆"、"行为异常"或"前后矛盾"时
3. 你自己感觉上下文混乱或不确定自己的配置时
4. 修改了任何核心文件（SOUL.md、MEMORY.md、AGENTS.md、openclaw.json）之后

## 自检清单（按顺序执行）

### 1. 配置一致性检查
**目的**：确保你对自己的认知与实际配置一致。
- 读取 `~/.openclaw/openclaw.json` 中 `agents.defaults.model` 和各 agent 的 `overrides.model`
- 读取 `AGENTS.md` 和 `MEMORY.md` 中关于模型的描述
- **对比**：Markdown 中描述的模型名称是否与 JSON 配置完全一致？
- 如果不一致：**修正 Markdown 文件**使其与 JSON 一致（JSON 是单一事实来源）
- 将修正记录写入今天的 `memory/YYYY-MM-DD.md`

### 2. 记忆健康检查
**目的**：防止 MEMORY.md 膨胀导致静默截断（20K 字符限制）。
- 检查 `MEMORY.md` 的字符数（使用 Coder 执行 `wc -c`）
- **阈值**：
  - < 15K 字符：✅ 健康
  - 15K-18K 字符：⚠️ 警告，考虑精简
  - > 18K 字符：🚨 危险，必须立即精简
- 检查 MEMORY.md 中是否混入了不属于"核心记忆"的内容（如完整的技能文档、代码片段、长篇教程）
- 如果发现膨胀内容：将其移到 `memory/archive/` 目录下的独立文件，在 MEMORY.md 中只留一行索引指针
- 记录操作到今天的日志

### 3. 工作区卫生检查
**目的**：保持文件系统整洁。
- 检查 `projects/` 目录下是否有名称相似但分裂的目录（如"大A"和"大"和"大 A"）
- 检查根目录是否有不属于标准布局的散落文件（.py、.sh 等应在 scripts/ 中）
- 检查是否有超过 7 天的 `.bak` 或 `.tmp` 文件
- 如果发现问题：**仅报告**，不自动合并目录（需用户确认）

### 4. 子 Agent 一致性检查
**目的**：确保子 Agent 的身份文件与主控一致。
- 检查 `~/.openclaw/workspace-coding/IDENTITY.md` 和 `~/.openclaw/workspace-researcher/IDENTITY.md` 是否为空或未初始化
- 检查子 Agent 的 `AGENTS.md` 中的模型描述是否与 openclaw.json 一致
- 如果不一致：修正子 Agent 的 Markdown 文件

### 5. 生成诊断报告
完成上述检查后，生成一份简洁的诊断摘要，写入 `memory/YYYY-MM-DD.md`，格式如下：

```markdown
## 🩺 工作区自检报告 (HH:MM)
- 配置一致性：✅ / ⚠️ [具体问题]
- MEMORY.md 健康：✅ XXXK/20K / ⚠️ 接近上限
- 工作区卫生：✅ / ⚠️ [具体问题]
- 子 Agent 一致性：✅ / ⚠️ [具体问题]
- 已自动修复：[列出修复项] / 无
- 需用户确认：[列出待确认项] / 无
```

## 修复权限规则
- **可以自动修复**：Markdown 文件中的模型描述不匹配、MEMORY.md 膨胀内容迁移、空的 IDENTITY.md 补全
- **需要用户确认**：目录合并、删除非 .bak/.tmp 文件、修改 SOUL.md 的核心规则
- **绝不修改**：openclaw.json（那是框架配置的单一事实来源，只能由用户或 CLI 修改）

## 设计原则
- 你不是在运行一个脚本，你是在**用自己的智能**审视自己的状态
- 每次自检都应该像一个有经验的运维工程师在做巡检，而不是机械地执行 grep
- 如果发现了检查清单之外的异常，也应该记录和报告
