# 文件交付相关教训

## 2026-03-09: 图片下载陷阱
**问题**：直接 curl/wget 搜索引擎返回的图片 URL，经常下载到 HTML/XML 错误页面。
**预防**：必须使用 `image-search-sender` 技能，发送前用 `file` 命令验证文件类型。

## 2026-03-09: delivery-queue 死信循环
**问题**：delivery-queue 中失败的消息无限重试，导致钉钉反复收到"图片加载失败"的占位图。
**预防**：`to` 字段必须是有效的钉钉 staffId，定期检查 delivery-queue/failed 目录。

## 2026-03-09: aliyun_code_interpreter 远程沙箱文件不在本地（严重！）
**问题**：使用 `aliyun_code_interpreter` 下载图片，文件只存在于远程沙箱，本地不存在。
**预防**：绝对禁止用 code_interpreter 下载文件到本地路径，下载图片必须用 image_search.py。

## 2026-03-09: 发送多张图片时只发了1张
**问题**：用户要求5张图片，只写了1个 `[DING:IMAGE]` 标签。
**预防**：用户要求几张就必须写几个标签，每张图片一个。

## 2026-03-09: 用户要求1张却发了5张
**问题**：用户说"发一张"，AI 却下载并发送了5张。
**预防**：仔细理解用户请求中的数量词，`--limit` 参数与用户要求一致，没说数量时默认1张。

## 2026-03-11: 走势图生成成功但未发送到钉钉群（Subagent 静默失败）
**问题**：图表在本地成功生成，但 Subagent 遇到 API rate limit 失败，用户什么都没收到。
**预防**：涉及钉钉图片发送的任务不委托 Subagent，执行口诀：生成 → 验证 → 发送 → 确认。
