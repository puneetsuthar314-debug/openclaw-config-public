## 二、提示词体系适配

### 2.1 系统级提示词（已存在）

**SOUL.md** — 身份与价值观
- 定义"丫丫"人设：干练、直接、有主见
- 核心价值观：诚实第一、先查后答、系统优先、执行导向
- 安全红线：删除/覆盖/外发需确认

**AGENTS.md** — 行为准则
- 任务分级标准（L1/L2/L3）
- 质量红线（禁止空洞内容、禁止为速度牺牲质量）
- 安全红线（删除/覆盖/外发/付费需确认）

### 2.3 用户级提示词模板（钉钉渠道适配）

```

【验收标准】{可验证的标准}
```

## 三、执行流程适配

### 3.1 标准工作流

```
┌─────────────────────────────────────────────────────────────┐
│  1. 任务接收 → 自动分级 (L1/L2/L3)                          │
│     - L1: 直接执行                                          │
│     - L2: 简要计划 → 执行                                   │
│     - L3: 写计划文件 → 分阶段 → 每阶段确认                  │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  2. 上下文准备                                               │
│     - 读取 memory/NOW.md (当前优先级)                       │
│     - 读取 memory/lessons.md (相关经验教训)                 │
│     - 读取 memory/infra.md (技术栈配置)                     │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  3. 执行 + 验证                                              │
│     - 每步执行前：确认理解正确                              │
│     - 每步执行后：自检产出是否完整可用                      │
│     - 涉及删除/覆盖/外发/付费：暂停等确认                   │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  4. 归档 + 进化                                              │
│     - 更新 memory/backlog.md (完成项标记)                   │
│     - 踩坑记录到 memory/lessons.md                          │
│     - 每日日志 memory/YYYY-MM-DD.md                         │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 钉钉渠道特殊规则

- **输出节奏**：遵循钉钉输出规范（详见 vault/reference/tools-manual.md）
  - 工具前短通知即时发送
  - 过程中不刷屏
  - 最终只发 1 条完整总结

- **媒体发送**：遵循 `dingtalk-send-media`
  - 图片/文件必须下载到本地后用 `[DING:IMAGE path="/abs/path"]` 发送
  - 禁止远程 URL 直接发送

- **定时任务**：遵循定时任务规范（使用 senderStaffId: manager20）
  - 使用用户 senderStaffId (manager20)

## 四、元提示词系统（可选高级功能）

### 4.1 α-提示词（生成器）

**用途**：生成新技能/提示词

**触发条件**：用户需要新能力时

**示例**：
```
你是一个技能生成器。根据以下需求生成一个 OpenClaw 技能：
- 需求：{用户描述}
- 现有技能：{当前 skills/ 列表}
- 输出格式：SKILL.md 标准格式
```

### 4.2 Ω-提示词（优化器）

**用途**：优化现有技能/提示词

**触发条件**：技能执行失败或被用户纠正时

**示例**：
```
你是一个技能优化器。分析以下失败案例并优化技能：
- 技能：{技能名称}
- 失败原因：{错误日志/用户反馈}
- 现有版本：{当前 SKILL.md 内容}
- 输出：优化后的 SKILL.md
```

### 4.3 递归进化 Loop

```
1. 执行任务 → 记录失败/纠正到 memory/lessons.md
2. 定期（每周）读取 lessons.md → 触发 Ω-提示词优化相关技能
3. 优化后的技能 → 更新 skills/ 目录
4. 新技能 → 用于后续任务 → 回到步骤 1
```

## 六、与 Vibe Coding 原版对比

| 维度 | Vibe Coding 原版 | OpenClaw 适配版 |
