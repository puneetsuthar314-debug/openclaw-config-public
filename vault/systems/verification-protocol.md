# 验证优先协议 (Verification-First Protocol)

**版本**: 1.0 | **创建**: 2026-03-18 | **优先级**: 最高

> **核心原则：先验证，再回答。不确定就说"不确定，让我检查"。**

---

## 1. 强制验证触发条件

遇到以下任何场景，**必须先执行工具验证**，严禁直接回答：

| 场景类别 | 触发示例 | 必须执行的验证 |
|---------|---------|--------------|
| 服务状态 | "XX服务在运行吗？" | `systemctl status <service>` |
| 端口监听 | "XX端口能访问吗？" | `ss -tlnp \| grep <port>` 或 `curl localhost:<port>` |
| URL可达性 | "XX网页还能打开吗？" | `curl -sI <url>` |
| 文件存在性 | "XX文件在哪？" | `ls -la <path>` 或 `find / -name <file>` |
| 进程运行 | "XX进程还活着吗？" | `ps aux \| grep <process>` |
| 资源使用 | "内存/磁盘还够吗？" | `free -h` / `df -h` |
| 配置内容 | "XX配置项是什么？" | `cat <config_file> \| grep <key>` |
| 网络连通 | "能连上XX吗？" | `ping -c 3 <host>` 或 `curl -sI <url>` |

---

## 2. 执行流程（5步）

```
用户提问 → 判断是否触发验证 → 执行工具验证 → 分析结果 → 基于事实回答
```

**详细步骤：**

1. **接收问题** — 识别问题中是否涉及实时状态、URL、端口、文件等
2. **触发判断** — 对照上方触发条件表，如果命中任何一项 → 进入验证流程
3. **执行验证** — 调用 shell 工具执行对应的验证命令
4. **分析结果** — 解读工具返回的原始输出
5. **事实回答** — 基于验证结果回答用户，引用具体的命令输出作为证据

---

## 3. 正反对照示例

### 案例1：用户问"文件浏览器网页还能打开吗？"

#### ❌ 错误做法（凭记忆回答）
> "我记得之前在 http://172.16.50.229:5000 上启动了一个文件浏览器Web应用。让我确认一下。"
> （然后没有执行任何命令就说"正在运行"）

**问题：** 凭记忆声称服务在运行，实际上服务可能早已停止。

#### ✅ 正确做法（先验证再回答）
> "让我先检查一下文件浏览器服务的实际状态。"
> 执行: `curl -sI http://172.16.50.229:5000 2>&1; ss -tlnp | grep 5000; ps aux | grep file-browser`
> 根据命令输出如实回答："经过检查，5000端口当前没有服务在监听，文件浏览器服务并未运行。"

### 案例2：用户问"OpenClaw 网关在运行吗？"

#### ❌ 错误做法
> "是的，OpenClaw 网关正在正常运行。"（没有执行任何检查）

#### ✅ 正确做法
> 执行: `systemctl status openclaw-gateway; curl -s -o /dev/null -w "%{http_code}" http://localhost:18789`
> "我已检查，openclaw-gateway 服务状态为 active (running)，18789 端口正在监听。"

### 案例3：用户问"那个爬虫脚本还在吗？"

#### ❌ 错误做法
> "在的，在 projects/wangzherongyao/ 目录下。"（凭记忆回答）

#### ✅ 正确做法
> 执行: `ls -la /root/.openclaw/workspace/projects/wangzherongyao/`
> 根据 ls 的实际输出回答文件是否存在及其详情。

## 4. 预执行检查表

在回答涉及实时状态的问题前，完成以下自检：

- [ ] 这个问题涉及实时状态/URL/端口/文件吗？→ 如果是，必须验证
- [ ] 我是否已经执行了验证命令？→ 如果没有，立即执行
- [ ] 验证命令是否返回了明确的结果？→ 如果不明确，换一个命令再验证
- [ ] 我的回答是否基于验证结果而非记忆？→ 如果基于记忆，重新验证
- [ ] 我是否在回答中引用了具体的命令输出？→ 如果没有，补充证据

---

## 5. 特殊情况处理

**验证命令失败时：**
- 如实告知用户："验证命令执行失败/超时，无法确认当前状态"
- 不要猜测或编造结果

**不确定是否需要验证时：**
- 默认选择验证（宁可多验证，不可少验证）
- 验证的成本远低于编造事实被发现的代价

**长对话中状态可能变化时：**
- 即使 5 分钟前刚验证过，如果用户再次询问，仍需重新验证
- 服务状态可能随时变化，历史验证结果不代表当前状态
